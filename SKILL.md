# Obsidian LLM Wiki Skill

用 LLM 持续维护 Obsidian 知识库。基于 raw/wiki/schema 三层架构。

## 前置条件

使用前，项目根目录必须有 `CLAUDE.md`（模式定义文件）。如果不存在，引导用户初始化：

1. 询问用户的知识库有哪些领域（如"读书笔记"、"AI"、"投资"）
2. 使用 [references/schema.md](references/schema.md) 作为模板，生成项目专属的 `CLAUDE.md`
3. 创建 `raw/` 和 `wiki/` 目录结构
4. 创建空的 `index.md` 和 `log.md`

## 模板

本 Skill 自带通用模板，位于 `assets/` 目录：
- `assets/wiki-page.md` — 通用 wiki 页面
- `assets/book-note.md` — 读书笔记
- `assets/meeting-note.md` — 会议记录
- `assets/tool-page.md` — 工具页面

**优先级：** 如果项目根目录有 `templates/` 目录，优先使用项目模板；否则使用 Skill 自带的 `assets/` 模板。

模板中的 `{{domain}}`、`{{date}}`、`{{title}}` 等占位符由 Claude 根据项目 CLAUDE.md 的配置填写。

## 命令

### /obsidian-llm-wiki ingest <source>

处理 raw/ 中的新来源，集成到 wiki。

步骤：
1. 确认来源文件在 `raw/` 对应目录中
2. 读取来源文件（PDF 使用 Read tool 的 pages 参数；含图片时按「图片文字辅助」阈值规则处理）
3. 与用户讨论关键要点
4. 在 `wiki/` 对应目录创建摘要页面：
   - 读取项目 CLAUDE.md 确定领域和路径映射
   - 选择合适的模板（从项目 templates/ 或 Skill assets/）
   - 添加 YAML frontmatter（根据 CLAUDE.md 中的规范）
   - 添加 inline tags（与 frontmatter 匹配）
   - 包含一句话摘要
   - 添加「相关」和「来源」部分
   - 确保排版符合规范（见下方「排版规范」）
5. （图片保护）读取将要更新的已有 wiki 页面，记录其中所有图片引用，包括文件名和尺寸参数
6. 更新与新内容相关的已有 wiki 页面，更新后逐一验证步骤 5 记录的图片引用仍然存在；如有丢失，立即恢复
7. 更新 `index.md`
8. 追加条目到 `log.md`，格式：`## [YYYY-MM-DD] ingest | <标题>`

### /obsidian-llm-wiki query <问题>

使用 wiki 回答问题。

步骤：
1. 读取项目 `CLAUDE.md` 了解架构
2. 读取 `index.md` 了解可用内容
3. 从索引定位相关 wiki 页面
4. 读取相关页面
5. 如果相关页面中引用了图片且问题涉及图片内容，按「图片文字辅助」阈值规则处理图片读取
6. 综合答案，引用页面 `[[标题]]`
7. 询问用户是否将答案归档为新 wiki 页面
8. 如果是，创建页面并更新 index.md 和 log.md

### /obsidian-llm-wiki lint

健康检查 wiki。

步骤：
1. 读取 `index.md`，验证每个列出的页面存在
2. 读取每个 wiki 页面，检查：
   - YAML frontmatter 存在且格式正确
   - frontmatter 中的 tags 与 inline tags 一致
   - 页面至少有 2 个出站 wiki 链接
   - 无断裂的 wiki 链接（链接到不存在的页面）
   - 同主题页面间无矛盾
   - 所有图片引用（`![[...]]`）指向的文件存在于 raw/ 或 wiki/ 目录中
   - 图片引用未被意外删除
3. 找出孤立页面（无其他页面的入链）
4. 找出值得拥有独立页面的内联提及
5. 检查过期内容（30+ 天未更新）
6. 检查图片引用质量：图片与周围内容是否相关、尺寸参数是否合理
7. 向用户报告发现，如发现可自动修复的问题（如缺失 frontmatter 字段、tags 不一致、断裂链接），询问用户是否执行修复
8. 如用户确认修复，执行修复后更新 `index.md`
9. 检查是否需要更新 `CLAUDE.md`（如发现新标签模式需注册、新领域需添加到注册表），如有则建议更新并等待用户确认
10. 追加 lint 报告到 `log.md`

### /obsidian-llm-wiki migrate

一次性迁移助手。用于将已有的 Obsidian 笔记迁移到 LLM Wiki 模式。

步骤：
1. 扫描项目目录，找出不在 raw/ 和 wiki/ 中的 .md 文件
2. 与用户确认每个文件属于哪个领域
3. 对每个文件：
   a. 读取文件内容
   b. 提取 inline tags
   c. 生成 YAML frontmatter（根据 CLAUDE.md 规范）
   d. 记录文件中所有已有图片引用
   e. 确定目标：raw/（源资料）或 wiki/（知识页面）
   f. 如目标为 wiki/，按「排版规范」调整格式
   g. 写入新位置，写入后验证所有图片引用完整保留
   h. 记录迁移日志到 `log.md`
4. 所有文件迁移完成后，重建 index.md

### /obsidian-llm-wiki index

从当前 wiki 状态重建 index.md。

步骤：
1. 扫描 wiki/ 目录下所有 .md 文件
2. 对每个文件，读取 frontmatter 和第一行内容
3. 按领域组织
4. 写入新的 index.md
5. 追加条目到 `log.md`，格式：`## [YYYY-MM-DD] index | 重建索引`

## 守则

- 绝不修改 `raw/` 下的文件
- 覆盖已有 wiki 内容前须确认
- 编辑 wiki 页面前，先记录页面中所有图片引用（`![[...]]` 模式）；编辑后验证所有图片引用仍然存在，如有丢失必须立即恢复
- 绝不因编辑、格式化、重写而删除页面中已有的图片引用（`![[文件名]]`、`![[文件名|尺寸]]`）
- log.md 条目 append-only
- 保持中文内容
- 不确定时询问用户
- 所有路径和领域从项目 CLAUDE.md 读取，不硬编码

## 排版规范

所有 wiki 页面应遵循以下排版标准。

### 标题层级

- 页面不使用 H1（`#`），标题由 frontmatter `title` 定义
- 正文从 H2（`##`）开始，层级严格递进不跳级
- 同一页面中 H2 之间保持语义平行

### 列表格式

- 无序列表统一使用 `-`（不混用 `*` 和 `+`）
- 嵌套缩进 2 个空格
- 任务列表使用 `- [ ]` 和 `- [x]`

### 表格

- 必须包含表头行和分隔行
- 仅在数据对比或多维度信息时使用，简单列表不转为表格

### Callout

使用 Obsidian callout 替代普通引用表达特殊语义：`> [!note]`、`> [!tip]`、`> [!warning]`、`> [!info]`、`> [!quote]`。一句话摘要仍用普通引用（`> 摘要`），不使用 callout。

### 代码块

- 行内代码用反引号包裹命令、文件名
- 代码块指定语言标识（` ```python `、` ```bash `）

### 图片

- 默认不指定尺寸（`![[图.png]]`），让 Obsidian 自适应
- 信息密度高的图表可限宽：`![[图.png|400]]`
- 不指定高度（保持纵横比）

## 图片文字辅助

wiki 页面中引用的图片（截图、图表、幻灯片）可使用 Read tool 直接读取获取文字信息。为控制资源消耗，按图片数量分级处理。

### 阈值规则

- **≤10 张图片**：主动读取所有图片中的文字，用于辅助摘要/回答/检查
- **>10 张图片**：默认不读取图片文字，但主动询问用户：
  - 是否需要读取全部图片
  - 是否指定特定区块的图片进行读取（如「只读 ## 核心图表 下的图片」）
  - 是否跳过纯装饰性插图（如封面、分隔线、头像等）

### 适用场景

- **ingest**：raw 来源含截图或扫描件时，读取图片获取文字辅助摘要
- **query**：用户提问涉及图片内容时，读取相关图片辅助回答
- **lint**：检查图片引用是否与上下文相关

### 精准控制

用户可通过以下方式指定读取范围：
- 按区块：`「读取 ## 核心图表 下的图片」` — 只读取该章节内的图片
- 按文件名：`「读取 截图01.png 和 流程图.png」` — 只读取指定文件
- 按类型：`「只读图表类图片，跳过截图」` — 根据上下文判断图片类型

### 使用方式

1. 统计当前任务涉及的图片总数
2. 根据阈值规则决定是否需要用户确认
3. 从图片引用 `![[文件名.png]]` 中提取文件路径
4. 根据 CLAUDE.md 领域注册表确定图片在 `raw/<领域>/` 中的完整路径
5. 使用 Read tool 读取图片（支持 PNG、JPG 等格式）

### 限制

- 读取结果作为辅助参考，不替代 raw 来源的文字内容
- 跳过明显为装饰用途的图片（分隔线、头像等）
